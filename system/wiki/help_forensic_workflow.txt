HELP FORENSIK (KONZEPT)
=======================

Stand: 2026-01-28

PHILOSOPHIE: HELP ALS WAHRHEIT
------------------------------
In BACH gilt das Prinzip: "Die Documentation (`docs/docs/docs/help/*.txt`) definiert das System."
Wenn das System anders verhaelt als die Doku, ist das ein Bug - entweder im Code oder in der Doku. Help Forensik ist der Prozess, diese Diskrepanz aufzuloesen.

WARUM "FORENSIK"?
-----------------
Wir gehen wie Ermittler vor. Wir trauen weder dem Code noch der Doku blind, sondern pruefen "Beweise":
- Funktioniert Befehl `bach --x` wirklich? (Tatort-Untersuchung)
- Steht er im Code? (Spurensicherung)
- Steht er in der Doku? (Aktenlage)

ABWEICHUNGS-KLASSEN
-------------------

### 1. Verbesserung (System > Doku)
Dies ist der haeufigste Fall in schnelllebigen Projekten. Features wurden "mal eben" eingebaut, aber nie dokumentiert.
**Aktion:** Doku nachziehen. "Fejern" des Features.

### 2. Zielzustand Fehlt (Doku > System)
Das "Vaporware"-Problem. Ein Feature wurde spezifiziert (und dokumentiert), aber nie oder nur teilweise gebaut.
**Aktion:** Ehrlich machen. Entweder als "Geplant" markieren oder Task erstellen.

### 3. Veraltet / Zombie (Doku != System)
Features, die es mal gab, die aber entfernt oder umbenannt wurden. Die Doku fuehrt User in die Irre.
**Aktion:** Doku bereinigen.

DER WERT VON VERSIEGELTEN AKTEN
-------------------------------
Eine gepruefte Help-Datei ist wie eine versiegelte Akte. Wir wissen, sie ist wahr (zum Zeitpunkt der Pruefung). Das gibt dem User (und der KI) Vertrauen.
Deshalb: Validierungs-Header (`# Zuletzt validiert: ...`) sind essenziell.
